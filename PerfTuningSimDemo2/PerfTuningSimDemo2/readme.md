
**从“线上卡顿”到一条清晰的 Leaks：一次可控泄漏的排查随笔**

做性能排查时，我最怕两种情况：一种是“感觉内存一直涨，但 Instruments 里又说不清楚”；另一种是“能复现卡顿，但每次栈不一样”。所以这次我特意在项目里做了一个“可控的坏例子”：在 Bad 模式下人为注入一段稳定可抓的泄漏，让 Leaks 给出一条确定的 backtrace，然后再用同一套流程对照 Optimized 模式，形成一条完整的排查闭环。

这篇随笔记录的就是这条闭环：**如何把“疑似内存问题”变成“可定位、可解释、可验证的泄漏修复”。**

---

## 1. 复现：让问题稳定发生，而不是“靠运气”

在 `PerfTuningSimDemo2` 里我准备了两种模式：

- **Bad**：更激进的内存分配 + 注入可控泄漏
- **Optimized**：降低分配规模 + 缩短 autorelease 生命周期 + 节流 + 不泄漏

复现步骤很简单：

1) `Mode = Bad`  
2) 点击 `Start Memory Churn`（或 `Start All`）  
3) 打开 Instruments → Leaks 录制  
4) 等待 Leaks 事件出现

这个阶段的目标不是“观察曲线”，而是**拿到第一条可信证据**：Leaks 直接指向某段代码，并且可重复出现。

---

## 2. 定位：Leaks 给出的不是“结论”，而是“证据链入口”

当 Leaks 报红时，最有价值的是下面两点：

- **Bytes Used**：告诉你泄漏的规模（你截图里已经累积到几十 MiB）
- **Source/Backtrace**：告诉你“谁分配的、为什么没回收”

这次 Leaks 很干脆：直接落到 `ViewController.maybeLeakMemoryOnce()`。这类命中是最理想的，因为它说明：

- Instruments 能追踪到分配点
- 代码路径稳定
- 问题可复现且可解释

---

## 3. 解释：为什么这段代码一定会泄漏？

核心代码（简化描述）是：

- 用 `UnsafeMutableRawPointer.allocate(byteCount:alignment:)` 分配一段原生内存
- `initializeMemory` 写入一些数据
- **没有** `deallocate()`（也没有 `free()` 或任何释放逻辑）

这类泄漏与 ARC/Swift 引用计数完全无关，因为它绕开了对象系统，属于“手动管理内存”范畴：

- ARC 只会管理引用类型对象的生命周期（以及桥接到 ObjC 的对象）
- `UnsafeMutableRawPointer` 分配的原生内存，需要你显式释放
- 如果你把指针丢掉，就等于永久失联，除非进程退出

为了避免一次性把进程顶到系统杀掉，我还加了“泄漏预算”：

- `leakBudgetBytes` 控制泄漏上限
- `leakedBytes` 记录累积量
- 达到预算就停止继续泄漏

这让泄漏既真实，又可控。

---

## 4. 为什么这种“故意泄漏”对性能排查有价值？

很多同学会问：既然是“故意写坏代码”，这有什么意义？

意义在于它是一个**训练用的标定点**：

- 你能用它验证：Leaks 工具是否配置正确、符号是否完整、采样是否能落到源码
- 你能用它训练团队：看到 Leaks 红点后，如何读 backtrace、如何判断泄漏类型（对象泄漏 vs 原生内存泄漏）
- 你能用它形成对照：Bad 模式必现，Optimized 模式应不再出现——这就是“修复可验证”的基本条件

实际线上问题往往更隐蔽：可能是 CoreFoundation create/copy 规则没遵守、可能是 malloc buffer 忘记 free、可能是 C++ new/delete 不匹配。训练好“从 Leaks 到源码”的路径，线上就不慌。

---

## 5. 修复思路：泄漏的修复不是“让曲线好看”，而是“归还所有权”

对于这种原生内存泄漏，修复一般有三种思路（从工程角度由简到复杂）：

1) **最直接：对称释放**  
   - 分配后在合适时机 `deallocate()`（或 `free()`）
   - 保证生命周期边界清晰

2) **用自动管理的容器封装所有权**  
   - 把指针包装成一个对象/结构体，在 `deinit` 里释放
   - 避免“忘记释放”变成编码习惯问题

3) **根本不需要手动内存时，改回 Foundation/Swift 容器**  
   - 能用 `Data` 就别裸指针
   - 能复用缓存就复用，不要频繁分配

在你的 demo 语境下，Bad 模式就是为了让 Leaks 命中；Optimized 模式则可以选择“完全不泄漏”，或者演示“分配—持有—释放”的正确姿势。

---

## 6. 验证：修复之后如何证明“真的好了”？

性能修复最怕“改完感觉好了”。正确的验证方式应该是：

- 用 **同一套复现步骤** 再跑一遍 Instruments Leaks
- 观察：
  - Leaks 事件是否不再出现
  - Allocations 的增长是否可控
  - resident 指标是否不再持续单调增长（注意：resident 不等于无泄漏，但“持续单调上升”往往是危险信号）

如果你能做到“同样操作路径、同样工作负载、Leads 从必现到不出现”，这就是一条合格的优化闭环证据。

---

## 7. 写在最后：把排查流程产品化

这次最满意的不是抓到泄漏本身，而是把过程变得“可重复”：

- 有明确的开关（Bad/Optimized）
- 有明确的指标（resident/cpu）
- 有明确的证据（Leaks 命中到源码）
- 有明确的对照（优化前后）

当你能把这种流程沉淀成团队习惯时，“线上卡顿排查”就不再是玄学，而是一套可以交付的工程能力。
